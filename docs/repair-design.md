# CafeClick 버그 분석 및 설계 문서

## 목표

기존 단일 파일 클릭커 게임의 핵심 루프를 안정화하고, README에 적힌 플레이 흐름이 실제 브라우저 화면과 항상 같은 상태를 가리키도록 만드는 것이 목표였습니다.

## 확인한 문제

- 자동 수입이 발생해도 마일스톤 문구가 바로 바뀌지 않았습니다.
- 보유 금액은 늘었지만 업그레이드 구매 가능 여부가 즉시 갱신되지 않았습니다.
- 사용자가 다시 클릭하거나 구매 버튼을 눌러야만 일부 UI가 뒤늦게 따라왔습니다.

## 원인

자동 수입 인터벌이 게임 상태는 변경했지만, 전체 UI를 다시 그리는 경로를 호출하지 않았습니다. 그 결과 숫자 카드, 마일스톤, 상점 버튼이 서로 다른 시점의 상태를 보여주는 구조가 됐습니다.

## 설계 방향

- `index.html` 단일 파일 구조는 유지합니다.
- 상태를 바꾸는 모든 동작이 하나의 전체 렌더링 함수로 모이게 합니다.
- 업그레이드 데이터와 마일스톤 데이터는 평범한 배열과 객체로 유지해 검증 가능성을 높입니다.
- 기존 README에서 설명하던 클릭 수입, 자동 수입, 업그레이드, 엔딩 흐름은 그대로 보존합니다.

## 데이터 흐름

1. 초기 진입 시 전체 UI를 렌더링합니다.
2. 클릭 시 `coins`, `total`, `cups`를 갱신한 뒤 전체 UI를 다시 그립니다.
3. 업그레이드 구매 시 비용 차감과 보유 수량 반영 후 전체 UI를 다시 그립니다.
4. 자동 수입 tick이 돌 때도 같은 전체 렌더링 경로를 호출해 마일스톤과 상점 상태를 즉시 동기화합니다.

## 검증 전략

- `tests/cafe-clicker.test.js`에서 Node `vm`으로 인라인 스크립트를 실행합니다.
- 가짜 DOM과 가짜 `localStorage`를 붙여 브라우저 없이도 게임 루프를 검증합니다.
- 초기 렌더, 클릭 수입, 업그레이드 구매, 가격 증가, 자동 수입 반영, 엔딩 흐름을 회귀 테스트로 확인합니다.
